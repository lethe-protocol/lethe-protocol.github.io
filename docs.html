Proof of Erasure Protocol

The cryptographic foundation of Lethe's verified destruction market.

Core Principle

The transaction implies erasure. A TEE-signed transaction submitted via roflEnsureAuthorizedOrigin() on Oasis Sapphire constitutes cryptographic proof that the code was received, processed, audited, and destroyed inside an Intel TDX enclave. The TEE identity is verified on-chain by the Sapphire runtime itself.

Protocol Steps

1. Bounty Creation

Requester deposits ROSE into LetheMarket.sol on Oasis Sapphire. Registers repo hash, audit configuration (trigger mode, scope), and GitHub repo info. All state is encrypted by default on Sapphire.

2. Polling & Trigger

ROFL TEE worker polls bountyCount() and getBounty() on a configurable interval. Evaluates trigger conditions (ON_CHANGE: new commits, PERIODIC: elapsed time). Skips bounties that don't meet trigger criteria.

3. Clone & Audit

TEE worker authenticates as a GitHub App (JWT + installation token, 1-hour expiry), clones the repository into ephemeral RAM storage, runs Semgrep static analysis. No network access during analysis. Creates a PR with findings on GitHub.

4. Erasure

NIST 800-88 compliant 3-pass destruction (0x00, 0xFF, 0x00) of all source code in memory. A keccak256 commitment hash is generated over the erasure operation. Source code directory is deleted. This step MUST complete before PoE submission.

5. Proof of Erasure

TEE worker submits submitAudit(bountyId, commitHash, poeHash, ruleVersion, payout) via appd socket. The transaction is signed inside the TEE. Sapphire verifies TEE identity via roflEnsureAuthorizedOrigin(). Escrow is released automatically.

Security Properties

  • Hardware attestation — Intel TDX memory encryption + ROFL attestation chain verified on Sapphire
  • Confidentiality — Code never leaves the TEE. Sapphire state is encrypted by default
  • Verifiable destruction — NIST 800-88 3-pass ensures data is unrecoverable
  • Sequentiality guarantee — Erasure MUST complete before on-chain PoE submission. This ordering is enforced in code
  • Ephemeral execution — RAM-only storage, no persistent disk. Each audit starts clean

Components

Oasis Sapphire

Confidential EVM. All contract state is encrypted by default. LetheMarket manages escrow, standing bounties, trigger modes, and repo registry. ReputationRegistry tracks performer scores with asymmetric penalties.

ROFL TEE Worker (Intel TDX)

Runtime OFf-chain Logic container running on real Intel TDX hardware. 4GB RAM, 2 vCPU, ephemeral RAM storage. Boots, polls for bounties, executes audits, submits PoE. Identity verified on-chain via roflEnsureAuthorizedOrigin().

appd Client

Hybrid query architecture. Public contract reads (bountyCount, getBounty) go via direct JSON-RPC. TEE-gated reads (getRepoInfo, getAuditConfig) go via appd CBOR-encoded queries. Write transactions go via appd UNIX socket with TEE-signed identity.

Semgrep Scanner

Static analysis engine running inside the TEE. No network access during scan. Extensible rulesets (currently: security, owasp, javascript, typescript, python, solidity).

GitHub App

Authenticates via JWT + installation token (1-hour expiry). Scoped to installed repos only. Clones private repositories, creates PRs with findings. Source code stays in TEE memory.

Erasure Engine

NIST 800-88 3-pass destruction (0x00, 0xFF, 0x00) of all cloned source code. Generates keccak256 commitment hash. Must complete before PoE submission. Defense-in-depth cleanup on failure paths.

Data Flow

1. Bounty — Requester deposits ROSE into LetheMarket. Registers repo info and audit config (trigger mode, scope, period).

2. Poll — TEE worker queries bountyCount() and getBounty() via JSON-RPC every 60 seconds. Evaluates trigger conditions (ON_CHANGE, PERIODIC).

3. Clone — GitHub App JWT generates installation token. Worker clones repo into ephemeral RAM storage inside TEE.

4. Audit — Semgrep runs inside TDX enclave. Findings formatted as PR body. Branch created, PR opened on GitHub.

5. Erase — NIST 800-88 3-pass destruction of source directory. Commitment hash generated.

6. Prove — submitAudit(bountyId, commitHash, poeHash, ruleVersion, payout) submitted via TEE-signed transaction. Sapphire verifies ROFL identity. Escrow released.

Vision Architecture

Open for Discussion — Help us build this

This is where Lethe is going. The core infrastructure — confidential escrow, TEE worker, Proof of Erasure — is deployed on Oasis Sapphire testnet. The designs below describe the full target architecture. We publish them openly so that security researchers, protocol engineers, and agent builders can challenge, refine, and contribute.

Discuss these proposals on GitHub Discussions.

1. Decentralized Performer Model

The current system runs a single ROFL TEE auditor on Oasis Sapphire. The target replaces this with a competitive market of independent performers, each running their own TEE environment.

Current:   Bounty → LetheMarket (Sapphire) → ROFL TEE auditor → On-chain PoE
                                                (single performer)

Target:    Bounty → LetheMarket (matching) → Performer's TEE → On-chain verification
                                               ↑
                                         Any TEE provider:
                                         • Oasis ROFL (TDX) — current
                                         • Self-hosted SGX/SEV
                                         • Phala Network
                                         • Marlin Oyster

Performers register by submitting a TEE attestation to the ReputationRegistry. The market matches bounties to performers via sealed-bid auction. Performers compete on price, speed, and reputation — not on privileged access.

2. Sealed-Bid Auction

When a requester submits an audit intent, qualified performers submit encrypted bids. After a designated window (e.g. 30 seconds), bids are revealed simultaneously. The market selects based on a combination of price, performer reputation, and specialization match.

This prevents front-running and ensures fair price discovery. The mechanism is not yet designed in detail — we welcome proposals.

3. Three-Tier LLM Verification Pipeline

The current PoC uses Semgrep for static analysis. The vision adds a layered verification architecture where untrusted input never reaches a privileged LLM directly.

Tier 1: Gatekeeper (no tools)
  Input:  Raw vulnerability report (untrusted)
  Output: Sanitized VLS (Vulnerability Logic Summary)
  Rule:   Strip injection attacks, structure into safe JSON

Tier 2: Analyst (sandboxed tools)
  Input:  Sanitized VLS only
  Tools:  execute_poc(), fetch_source(), run_static_analysis()
  Env:    TEE or sandbox — no persistent storage, no network
  Output: Verification report (verified / debunked / inconclusive)

Tier 3: Judge (no tools)
  Input:  Analyst report + VLS + metadata
  Output: accept / reject / request_more_info
  Rule:   confidence ≥ 0.8 → auto-settle

The key principle: a tool-bearing LLM never touches untrusted input directly. This prevents prompt injection from compromising the verification pipeline.

4. Asymmetric Reputation System

Reputation must make betrayal structurally unprofitable. The proposed formula:

Success:  S_new = S_old + (0.10 × (100 - S_old))       // slow growth
Failure:  S_new = S_old - (0.25 × S_old × 1.6^streak)  // exponential pain

Loss/gain ratio: 2.5×
Consecutive failure: exponential penalty
  1 failure:  -25%
  2 failures: -40%
  3 failures: -64%

Circuit breaker:
  Score ≥ 50: ACTIVE
  Score 30-49: WARNING (reduced privileges)
  Score < 30: SUSPENDED (blocked from market)

Reputation is bound to a Soulbound Token (SBT) — non-transferable. A performer's track record stays with them. Forking the market doesn't fork the reputation.

5. Dispute Escalation

Three levels, with increasing cost borne by the losing party:

Level 1 — Self-Verification

Requester verifies findings independently. Cost: free. Resolves most disputes.

Level 2 — Single Arbiter

A neutral arbiter agent reviews the case inside a fresh TEE. Cost: 5% of transaction value, paid by loser.

Level 3 — Three-Judge Panel

Majority vote from three independent arbiter agents. Cost: 15% of transaction value, paid by loser. Final and binding.

6. Domain Expansion

The escrow layer is designed to be domain-agnostic. The verification logic is a plugin. Expanding to new domains means writing a new verification plugin — not modifying the market infrastructure.

Phase 1: Static analysis audit — Semgrep inside TEE (current)
Phase 2: LLM-powered deep analysis — Ollama inside TEE (next)
Phase 3: General code review
Phase 4: Data integrity verification
Phase 5: AI model security (adversarial robustness)

Lower layer (domain-agnostic):
  Confidential escrow, matching, settlement, reputation

Upper layer (pluggable):
  Verification protocol per domain

7. TEE Limitations — Honest Assessment

TEE hardware has been breached before (Foreshadow 2018, Plundervolt 2019, SGAxe 2020, AEPIC Leak 2022). Lethe's core premise — "information exists only inside the TEE" — is only as strong as the TEE itself.

Mitigations on the roadmap:

  • Multi-vendor TEE — Mix Intel SGX + AMD SEV + ARM TrustZone. A single vendor vulnerability doesn't compromise the entire market.
  • ZK transition — Long-term, replace hardware trust with mathematical proof. Currently impractical for LLM inference cost, but the field is advancing rapidly.
  • Continuous self-audit — Use the market's own agents to audit the market's TEE infrastructure.

We believe TEE-based erasure is the best available primitive today. It is not perfect. We say this openly because security communities rightly distrust projects that claim invulnerability.

These designs are open proposals. Challenge them.

LetheMarket — Public Functions

VIEW bountyCount() → uint256

Total number of bounties created.

VIEW getBounty(uint256 id) → Bounty

Returns bounty details: requester, amount, repoHash, deadline, standing flag, state (Open/Completed/Cancelled), auditCount.

WRITE createBounty(bytes32 repoHash, uint256 deadline, bool standing)

Create a new bounty with ROSE deposit. Standing bounties fund multiple audits from a single pool.

LetheMarket — TEE-Gated Functions

These require roflEnsureAuthorizedOrigin() — only callable from inside the registered ROFL TEE.

TEE getRepoInfo(uint256 id) → (string owner, string repo, uint256 installationId)

Repository details for cloning. Encrypted in Sapphire state, only accessible from TEE.

TEE getAuditConfig(uint256 id) → (uint8 triggerMode, uint8 scopeMode, uint8 toolMode, uint256 periodDays)

Audit configuration. Trigger modes: ON_CHANGE (0x01), PERIODIC (0x08). Combinable via bitmask.

TEE submitAudit(uint256 bountyId, bytes32 commitHash, bytes32 poeHash, uint256 ruleVersion, uint256 payout)

Submit Proof of Erasure. Only callable by the registered ROFL app. Verifies TEE identity, records audit metadata, releases payout.
